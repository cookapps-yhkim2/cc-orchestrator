# unity-sprint

스프린트 계약을 협상하고 관리하는 스킬입니다.
Planner와 Implementer 간 "뭘 만들지, 어떻게 검증할지"를 합의합니다.

## 트리거

- "스프린트 계획 세워줘"
- "다음에 뭐 만들지 정하자"
- "스프린트 시작하자"
- "새 스프린트"

## 전제 조건

- `.unity-orc/` 디렉토리가 존재해야 합니다. 없으면 `unity-init` 스킬을 먼저 실행하세요.
- `.unity-orc/feature_list.json`과 `unity-progress.md`가 있어야 합니다.

## 스프린트 계약 패턴

> 핵심: 구현 전에 "무엇을 만들지, 어떻게 검증할지"를 협상합니다.
> 모호한 요구사항을 구체적인 체크리스트로 변환하는 것이 목적입니다.

### 계약이 포함해야 할 것
1. **스프린트 목표**: 한 줄 요약
2. **기능 목록**: 이번 스프린트에서 구현할 기능 2-4개
3. **각 기능의 합격 조건**: Evaluator가 검증할 체크리스트
4. **구현 순서**: 의존성을 고려한 기능 구현 순서
5. **미결 사항**: 추가 확인이 필요한 것들

### 계약에 포함하면 안 되는 것
- 구체적인 코드 구현 방법 (Implementer의 재량)
- 지나치게 세부적인 UI 레이아웃 (프로토타입 단계에서는)
- 테스트 불가능한 합격 조건 ("보기 좋게" → ❌)

## 수행 작업

### 1단계: 현황 파악
1. `.unity-orc/unity-progress.md`를 읽어 현재 상태를 파악합니다.
2. `.unity-orc/feature_list.json`를 읽어 기존 기능과 완료 상태를 확인합니다.
3. 이전 스프린트의 평가 리포트(`.unity-orc/evaluations/`)를 확인합니다.
4. `open-questions.md`에 미해결 사항이 있는지 확인합니다.

### 2단계: 요구사항 수집
사용자에게 다음을 질문합니다:
- **이번 스프린트에서 만들고 싶은 것은?** (기능 목록)
- **우선순위가 높은 것은?** (순서 정하기)
- **특별히 신경 써야 할 제약 조건은?** (성능, 플랫폼, 에셋 등)

### 3단계: 스프린트 계약 초안 작성
`references/sprint-contract-template.md`를 참고하여 초안을 작성합니다.

사용자에게 초안을 보여주고 피드백을 받습니다:
- 기능이 너무 많은가? 줄일까?
- 합격 조건이 현실적인가?
- 빠진 기능이 있는가?

### 4단계: 합의 및 저장
사용자 승인 후:
1. `.unity-orc/plans/sprint-{번호}.md`에 저장합니다.
2. `.unity-orc/feature_list.json`에 새 기능을 추가합니다 (`passes: null`).
3. `.unity-orc/unity-progress.md`의 "현재 스프린트" 섹션을 업데이트합니다.

### 5단계: Planner Agent 호출
스프린트 계약이 확정되면, Planner Agent를 호출하여 각 기능의 상세 스펙을 작성하도록 합니다.

## 스프린트 번호 규칙

- 첫 번째 스프린트는 `sprint-001`
- 번호는 항상 3자리 (`001`, `002`, ..., `999`)
- 이전 스프린트가 완료(합격)되지 않아도 새 스프린트를 시작할 수 있지만, 사용자에게 경고합니다.

## 스프린트 규모 가이드

| 규모 | 기능 수 | 적합한 상황 |
|------|--------|-----------|
| 소형 | 1-2개 | 빠른 프로토타입, 버그 수정 |
| 중형 | 2-4개 | 일반적인 개발 사이클 |
| 대형 | 4-6개 | 큰 시스템 구축 (비추천 — 쪼개세요) |

## 완료 조건

- [ ] 사용자와 스프린트 범위를 합의함
- [ ] `.unity-orc/plans/sprint-{번호}.md` 파일이 생성됨
- [ ] `feature_list.json`에 새 기능이 추가됨
- [ ] `unity-progress.md`가 업데이트됨
